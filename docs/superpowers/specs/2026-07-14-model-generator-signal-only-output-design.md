# 模型生成器仅信号触发输出设计

## 目标

模型生成器中的模型只作为运行时生成模板。用户把普通模型或内置基础网格拖入属性面板后，编辑态不直接显示该模型；只有设备遥测、普通货物快照或 `warehouseFlow` 状态需要货物时才创建派生输出，信号失效或流程结束后销毁。

2026-07-15 起，模型生成器进一步收敛为场景级全局自动模型管理器：POI 库继续保留“模型生成器”卡片，可点击或拖到场景任意位置；实体 Transform 只控制编辑态青色配置标记，不是任何货物生成点。

## 兼容策略

- 保留现有 `ModelGeneratorComponent` 数据结构和场景文件版本 `1`。
- `defaultTarget` 字段表示“共享生成模板”，不代表编辑态默认显示，也不代表生成器标记位置处的无条件输出。
- `rule.target` 继续作为规则专属覆盖模型；规则命中但未配置专属模型时使用 `defaultTarget`。
- 新建场景只允许一个有效模型生成器；重复点击、拖入、复制、粘贴和阵列会选中已有生成器或拦截副本。
- 旧场景无需迁移。旧场景中的 `defaultTarget` 自动按共享模板新语义运行；若旧场景存在多个模型生成器，运行时只启用 `scene.entityIds` 中第一个，其余仅显示编辑标记并记录诊断。

## 编辑态行为

- 生成器始终只显示青色可拾取线框标记。
- 生成器 Transform 只影响该青色配置标记的位置、旋转和缩放，方便用户把配置入口放在场景任意处；它不参与普通 Conveyor、普通 Stacker 或 `warehouseFlow` 的货物坐标计算。
- `defaultTarget` 和 `rule.target` 都只是配置快照，不创建 Babylon Mesh、模型容器或脚本实例。
- Gizmo、显隐、锁定、聚焦和包围盒继续以青色配置标记为准。
- 自动货物不可拾取、不进入 Hierarchy、不写入场景文件或撤销历史。

## 运行态行为

1. Runtime 读取当前场景 `entityIds` 中第一个模型生成器作为全局生成器。
2. 普通 Conveyor、普通 Stacker 和 `warehouseFlow` 为每个有效货物快照独立解析规则，因此不同设备和条码可以同时使用不同模板。
3. 规则属性名先支持 `sourceId`、`deviceType`、`assetCode`；其余属性名读取 `snapshot.fields`。
4. 规则按列表顺序匹配。命中后先计算候选目标 `rule.target ?? defaultTarget`；候选目标为空属于不完整规则，忽略并继续后续规则。
5. 扫描后没有规则命中时使用 `defaultTarget`。
6. 普通 Conveyor/Stacker 没有生成器、没有可用模板或导入模型最终加载失败时，继续显示原有默认 Box，保证旧场景可视化不退化。
7. `warehouseFlow` 没有可用模板或导入模型最终加载失败时保持 fail-closed，不创建仓储货物。
8. 模板规则变化时按当前设备或库位支撑点重新解析并替换可视输出，不改变设备、货叉、locator 或仓储状态机决定的位置。
9. 停止预览、货物信号消失、设备实例删除、生成器配置切换、场景切换和 Runtime 销毁时统一释放 Mesh、材质、模型容器、贴图和外置脚本。

## 普通设备示例

- 共享生成模板：拖入一个货物模型。
- 规则一：`sourceId` 等于 `default`，规则覆盖模型可留空。
- 规则二：`assetCode` 等于 `DDJ2`，规则覆盖模型可配置堆垛机专用托盘。
- 普通 Conveyor 有货时，货物根节点放在输送面支撑点；普通 Stacker 有货时，货物根节点放在货叉支撑点或目标 locator 支撑点。
- 生成器标记 Transform 不改变上述位置；共享模板为空或最终加载失败时，普通设备继续显示旧版默认 Box。

## warehouseFlow 示例

- `warehouseFlow` 通过三条稳定 binding ID 引用入库 Conveyor、Stacker 和出库 Conveyor。
- 入库触发时复用全局生成器的规则/共享模板决定货物外观，但位置由仓储协调器决定：1004 输送面、DDJ2 货叉、目标 locator、1005 输送面。
- 无可用模板时不创建仓储货物，保持 fail-closed；不会回退默认 Box，以免仓储状态机误以为真实货物已存在。

## Inspector 文案

- 默认生成对象改为“共享生成模板”。
- 规则生成对象改为“规则覆盖模型”，并标注可选。
- 生成器提示必须强调“标记位置仅用于编辑配置，不影响自动模型位置”。
- 空规则提示改为“暂无生成规则，将使用共享生成模板；共享模板为空时普通设备回退默认 Box，warehouseFlow 不创建货物”。
- TTL 提示改为“数据超时后清理当前运行时货物或等待设备/仓储流程恢复”。

## 2026-07-15 变更记录

- 保留 POI 库“模型生成器”点击和拖拽入口，但 Transform 仅控制编辑态青色配置标记。
- 废弃“生成器标记坐标等于货物坐标”的旧语义；普通 Conveyor、普通 Stacker 和 `warehouseFlow` 的货物位置分别来自输送面、货叉、locator 或仓储状态机。
- 废弃“通过多个生成器实体配置多条线”的旧语义；新建、复制、粘贴和阵列都不得产生第二个有效生成器。
- 全场景按 `scene.entityIds` 第一个生成器统一管理模板/规则；旧场景多生成器只启用第一个。
- 普通设备无生成器、无模板或模板加载失败时回退默认 Box；`warehouseFlow` 无模板时继续 fail-closed。

## 验证标准

- 配置共享模板后，编辑态仍只显示青色线框。
- 拖动模型生成器标记后，普通 Conveyor、普通 Stacker 和仓储流货物位置不发生变化。
- 点击、拖拽、复制、粘贴或阵列不能创建第二个有效生成器。
- 同一时刻多个设备/条码可生成独立模型，规则可按设备身份或字段选择不同模板。
- 无生成器或无模板时普通设备仍显示原默认 Box；`warehouseFlow` 继续在无模板时拒绝启动。
- 停止运行预览后所有自动货物资源释放，编辑态仅显示 POI 标记。
- `typecheck`、`build` 和 `git diff --check` 通过。
