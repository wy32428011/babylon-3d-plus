# Scene 视角拖拽忽略模型拾取设计

**日期：** 2026-07-16

## 背景与根因

Scene View 的 ArcRotateCamera 没有启用模型碰撞，正常大幅拖拽可以直接从模型表面开始旋转视角。当前冲突来自点击选择判定：左键释放时仅按指针总位移是否大于 `4 px` 判断是否为拖拽，因此 `1–4 px` 的轻微视角拖拽已经改变相机位姿，却仍会继续拾取鼠标下方模型，表现为模型“阻挡”或抢占视角操作。

## 目标

- 只要本次左键交互实际改变了相机位姿，就按视角拖拽处理，不执行模型拾取。
- 纯单击模型仍正常选中，纯单击空白仍正常清空选择。
- 保留现有像素容差，避免未引起相机变化的轻微鼠标抖动破坏单击选择。
- 不改变模型 `isPickable`、Gizmo、F 聚焦、运行预览、相机保存/复位和场景文件结构。

## 方案比较

### 方案 A：锁存相机输入并以位姿变化兜底（采用）

Pointer down 时记录相机位姿；每次 pointer move 结束后通过微任务读取 Babylon 已累计的旋转/平移输入，一旦确认本次会话驱动过相机便永久标记为视角拖拽。Pointer up 再用相机位姿变化做兜底，随后才考虑像素容差和模型拾取。

- 优点：不依赖最终净位移，拖出后回到原坐标仍保持拖拽语义。
- 优点：覆盖快速同帧微拖拽，避免 alpha/beta 尚未刷新时误判为点击。
- 优点：保留现有单击容差；没有进入 Babylon 相机输入的轻微抖动仍可按点击处理。
- 风险：依赖当前 Babylon 9.12 的公开 movement 累计输入字段，后续升级 Babylon 时需随类型检查复核。

### 方案 B：把点击容差从 4 px 调低

实现最短，但仍是经验阈值，不同 DPI、灵敏度和设备下可能再次出现同类问题。

### 方案 C：取消模型拾取或调整相机鼠标按键

会破坏模型单击选中或改变既有操作习惯，不采用。

## 数据流

1. `SceneViewPanel.handleCanvasPointerDown()` 保存 pointerId、指针起点和 `viewport.getCameraPose()`。
2. ArcRotateCamera 继续通过现有 `attachControl()` 处理旋转、平移和缩放。
3. `handleCanvasPointerMove()` 在事件结束后的微任务中读取 Babylon movement 累计输入，并锁存 `cameraDragged`。
4. `handleCanvasPointerUp()` 先检查 `cameraDragged`，再用交互前后的相机位姿变化兜底。
5. 只有本次会话没有驱动相机时，才继续执行既有像素容差、Gizmo 避让和模型拾取逻辑。

## 验收标准

- 从模型表面开始进行 1–4 px 轻微拖拽时，视角可以移动且不会选中该模型。
- 从模型表面进行明显拖拽时，视角继续正常移动且不会选中模型。
- 不移动鼠标直接单击模型时，模型仍正常选中并显示 Gizmo。
- 单击空白区域仍清空选择。
- F 聚焦、相机保存/复位、运行预览相机和拖拽资源落点不受影响。
